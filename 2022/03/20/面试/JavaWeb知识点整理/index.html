1 Servlet

Servlet含义是服务器端的小程序，Servlet=Server+applet：

  • Server：服务器
  • applet：小程序

Servlet事实上就是Web应用中服务器端的控制器，负责处理请求、协调调度。

1.1 Servlet继承关系

继承关系：

  • 接口javax.servlet.Servlet
    • -> 抽象类javax.servlet.GenericServlet
      • -> 抽象子类javax.servlet.http.HttpServlet

javax.servlet.Servlet ：

  • 初始化方法：void init(config)
  • 服务方法：void service(request, response)
  • 销毁方法：void destroy()

javax.servlet.GenericServlet中，void service(request, response)仍然是抽象方法。

javax.servlet.http.HttpServlet中，void service(request, response)不再是抽象方法。

void service(request, response)：

  1. String method = req.getMethod();，获取请求方式；
  2. 各种if进行判断，根据请求方式的不同去调用不同的do方法；
  3. 各种do方法都差不多，一般自己定义Servlet的时候要进行重写。

当有请求过来时，service方法会自动响应（其实是tomcat容器调用），在HttpServlet中分析请求方式，是get，post等等，然后决定调用哪个do方法，如果没有重写，调用HttpServlet中的相应do方法会返回405错误。

1.2 Servlet生命周期

生命周期：从出生到死亡的过程，对应Servlet中的三个方法：init()，service()，destroy()。

默认情况下：

  1. 第一次接收请求时，这个Servlet会进行实例化（调用构造方法，Tomcat底层使用反射）、初始化（调用init()）、然后服务（调用service()）。
  2. 第二次请求开始，每一次都是服务。
  3. 容器关闭时，其中所有的Servlet实例会被销毁，调用destroy()方法。

注意：

  • Tomcat对于一个Servlet只会创建一个实例，所有请求都由这个实例去响应。
  • 默认情况下第一次请求时Tomcat才会去实例化，初始化。好处是可以提高系统启动速度，但是对于第一次请求的用户响应时间会很长。

Servelet的初始化时机：

  • 默认情况是第一次请求时实例化，初始化。
  • 可以通过<load-on-startup>标签来设置Tomcat启动过程中Servlet启动的先后顺序，数字越小启动越靠前。这样Tomcat启动时间会变慢，但是能提高给用户的响应速度。

注意：Servlet在容器是单例的，线程不安全的。

比喻：两个人先后去同一间厕所上厕所，前一个人提前观测发现还有卫生纸剩余，准备过会进去，这个期间第二个人已经进去把厕纸用完，第一个人进去蹲下的时候就傻了。

因此，尽量不要在Servlet中定义成员变量，如果不得不定义，则尽量不要去修改成员变量的值，也不要依靠它的值来做逻辑判断等操作。

2 HTTP协议

HTTP：Hyper Text Transfer Protocol，超文本传输协议。

  • HTTP协议是无状态的。
  • HTTP包含两个部分：请求和响应。
  • 作用是确定了请求和响应数据的格式。浏览器发送给服务器的数据：请求报文；服务器返回给浏览器的数据：响应报文。

2.1 HTTP请求报文

主要包括三个部分：请求行，请求消息头，请求体。

请求行包含：

  1. 请求的方式；
  2. 请求的URL；
  3. HTTP协议的版本（一般是HTTP1.1）。

请求消息头包含很多客户端想要告诉服务器的信息。

  • 作用：通过具体的参数对本次请求进行详细的说明。
  • 格式：键值对，键和值之间使用冒号隔开。

比较重要的请求消息头：

请求体有三种情况：

  1. get方式没有请求体，但是有一个queryString，以键值对用&拼接的形式紧跟在URL后面。
  2. post方式，有请求体Form Data，也是键值对用&拼接。
  3. json格式，有请求体Request Payload。

2.2 HTTP响应报文

也包含三个部分：响应行，响应消息头，响应体。

响应行包含：

  1. 协议；
  2. 响应状态码；
  3. 响应状态。

响应头包含了服务器发送给浏览器的信息，比如内容的媒体类型、编码、长度等。

响应体包含了响应的实际内容，比如请求某一个html页面响应的内容就是html文档。

2.3 响应状态码

作用：以编码的形式告诉浏览器当前请求处理的结果。

3 会话session

3.1 session的作用和常用API

HTTP是无状态的：服务器没法知道用户上一次做了什么。比如服务器无法判断两次请求是同一个客户端还是不同的客户端发过来的。

比如，第一次请求添加商品到购物车，第二次请求是结账，如果服务器无法区分两次请求来自同一个用户，就会出现混乱。

-> 解决办法：会话跟踪结束

  • 客户端第一次发请求给服务器，服务器获取session，获取不到则创建新的，然后响应给客户端；
  • 下一次客户端给服务器发请求时，会把sessionID带给服务器，服务器就能获取到然后判断这一次请求和上一次来自同一个客户端。

常用的API：

  • request.getSession() -> 获取当前的会话，没有则创建新的会话。
  • request.getSession(true) -> 效果和不带参数相同。
  • request.getSession(false) -> 获取当前会话，没有时返回null，不会创建新的会话。
  • session.getId() -> 获取sessionID。
  • session.isNew() -> 获取当前session是否是新的。
  • session.getMaxInactiveInterval() -> session的非激活间隔时长，默认1800s。
  • session.setMaxInactiveInterval()
  • session.invalidate() -> 强制让会话立即失效。
  • …

3.2 session保存作用域

session保存作用域是和具体的某一个session对应的

常用的API：

  • void session.setAttribute(k,v)
  • Object session.getAttribute(k)
  • void removeAttribute(k)

3.3 保存作用域的类型

原始保存作用域可以认为有四个：page（页面级别，现在几乎不用） , request（一次请求响应范围） , session（一次会话范围） , application（整个应用程序范围）。

  1. request：一次请求响应范围

  1. session：一次会话范围有效

  1. application： 一次应用程序范围有效

4 Cookie

Cookie是浏览器访问服务器后，服务器传给浏览器的一段数据，浏览器需要保存这段数据，此后每次浏览器访问该服务器，都必须带上这段数据。

比如客户端会保存sessionID，在下一次发送请求时带上。

Cookie的使用流程：

  1. 创建Cookie对象。
  2. 在客户端保存：response.addCookie(cookie)
  3. 设置Cookie：
    • cookie.setMaxAge(60) ：设置cookie的有效时长是60秒
    • cookie.setDomain(pattern)
    • cookie.setPath(uri)
  4. Cookie的应用：
    • 记住用户名和密码十天：setMaxAge(60 * 60 * 24 * 10)；
    • 十天免登录。

5 服务器内部转发与客户端重定向

服务器内部转发：request.getRequestDispatcher("...").forward(request, response);

  • 只有一次请求响应的过程，对于客户端而言，内部经过了多少次转发，客户端是不知道的。
  • 地址栏没有变化

客户端重定向：response.sendRedirect("...");

  • 两次请求响应的过程。客户端肯定知道请求URL有变化。
  • 地址栏有变化。

6 过滤器Filter

6.1 Filter介绍和使用

Filter的三要素：

  1. 拦截：过滤器之所以能够对请求进行预处理，关键是对请求进行拦截，把请求拦截下来才能够做后续的操作。而且对于一个具体的过滤器，它必须明确它要拦截的请求，而不是所有请求都拦截。
  2. 过滤：根据业务功能实际的需求，看看在把请求拦截到之后，需要做什么检查或什么操作，写对应的代码即可。
  3. 放行：过滤器完成自己的任务或者是检测到当前请求符合过滤规则，那么可以将请求放行。所谓放行，就是让请求继续去访问它原本要访问的资源。

Filter也属于Servlet规范。

Filter开发步骤：

  1. 新建类实现Filter接口，然后实现其中的三个方法：init、doFilter、destroy。
  2. 配置Filter，可以用注解@WebFilter，也可以使用xml文件 <filter>和<filter-mapping>。

注意：Filter在配置时，和servlet一样，也可以配置通配符，例如@WebFilter("*.do")表示拦截所有以.do结尾的请求。

6.2 过滤器链

多个Filter的拦截范围如果存在重合部分，那么这些Filter会形成Filter链，浏览器请求重合部分对应的目标资源时，会依次经过Filter链中的每一个Filter。

  • 执行的顺序依次是： A B C demo03 C2 B2 A2。
  • 如果采取的是注解的方式进行配置，那么过滤器链的拦截顺序是按照全类名的先后顺序排序的。
  • 如果采取的是xml的方式进行配置，那么按照配置的先后顺序进行排序。

7 监听器Listener

7.1 观察者模式

观察者模式由观察者和被观察者组成，比如JavaScript中对鼠标点击事件的监听。

  • 观察者：监控『被观察者』的行为，一旦发现『被观察者』触发了事件，就会调用事先准备好的方法执行操作。
  • 被观察者：『被观察者』一旦触发了被监控的事件，就会被『观察者』发现。

7.2 Listener介绍与分类

监听器：专门用于对其他对象身上发生的事件或状态改变进行监听和相应处理的对象，当被监视的对象发生情况时，立即采取相应的行动。

Servlet监听器：Servlet规范中定义的一种特殊类，它用于监听Web应用程序中的ServletContext，HttpSession和HttpServletRequest等域对象的创建与销毁事件，以及监听这些域对象中的属性发生修改的事件。

分类：

  1. 域对象监听器：

    • ServletContextListener - 监听ServletContext对象的创建和销毁的过程
    • HttpSessionListener - 监听HttpSession对象的创建和销毁的过程
    • ServletRequestListener - 监听ServletRequest对象的创建和销毁的过程

  2. 域对象的属性域监听器

    • ServletContextAttributeListener - 监听ServletContext的保存作用域的改动(add,remove,replace)
    • HttpSessionAttributeListener - 监听HttpSession的保存作用域的改动(add,remove,replace)
    • ServletRequestAttributeListener - 监听ServletRequest的保存作用域的改动(add,remove,replace)

  3. Session域中数据的监听器

    • HttpSessionBindingListener - 监听某个对象在Session域中的创建与移除
    • HttpSessionActivationListener - 监听某个对象在Session域中的序列化和反序列化
